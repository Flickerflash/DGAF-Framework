> **Four mandatory pillars applied to every artifact before registry sign-off or external publication. ≥ 3/4 per pillar across N ≥ 3 independent runs required. Failure on any pillar blocks promotion.**

---

## Rationale

Artifacts that are technically correct can still fail on epistemic, ethical, or historical grounds. A mathematically coherent claim can be epistemically dishonest (overstated confidence), non-violative on its surface but harmful in context, or historically sequential in isolation but contradicting established prior art in the registry. The 1-1-1-1 Gate enforces four orthogonal quality dimensions simultaneously — no single strong pillar can compensate for a failed one. This is the primary normative filter before any artifact enters the read-only registry or crosses an external publication boundary.

The gate is named 1-1-1-1 because each pillar scores on a binary pass/fail per run (1 = pass, 0 = fail), and the aggregate threshold is ≥ 3/4 passes per pillar over N ≥ 3 runs — giving statistical confidence rather than a single-point evaluation.

## The Four Pillars

| Pillar | Label | Question | Failure Mode |
|--------|-------|----------|--------------|
| **1** | Mathematically Coherent | Is every quantitative claim internally consistent and derivable? | Numeric contradiction, undefined variable, broken formula |
| **2** | Epistemically Honest | Are confidence levels accurate? Is uncertainty disclosed? | Overstatement, hidden assumptions, false precision |
| **3** | Non-Violative | Does the artifact respect AXIS declarations and ethical constraints? | ANDROMEDA-AXIS breach, OWASP Agentic Top 10 exposure, sovereignty violation |
| **4** | Historically Sequential | Is the artifact consistent with the established registry provenance chain? | Contradicts prior certified artifact, breaks changelog lineage, orphaned reference |

## Passing State

All four pillars pass at ≥ 3/4 across N ≥ 3 runs. Artifact cleared for registry promotion.

```json
{
  "gate": "GATE-1111",
  "status": "PASS",
  "agent": "Apogee",
  "runs": 3,
  "pillars": {
    "mathematically_coherent": { "score": "3/3", "pass": true },
    "epistemically_honest":    { "score": "3/3", "pass": true },
    "non_violative":           { "score": "3/3", "pass": true },
    "historically_sequential": { "score": "3/3", "pass": true }
  },
```

## Failing State

Any pillar scoring < 3/4 across N ≥ 3 runs blocks promotion. Artifact quarantined to `docs/drafts/`.

```json
{
  "gate": "GATE-1111",
  "status": "FAIL",
  "agent": "Apogee",
  "runs": 3,
  "pillars": {
    "mathematically_coherent": { "score": "3/3", "pass": true },
    "epistemically_honest":    { "score": "2/3", "pass": false },
    "non_violative":           { "score": "3/3", "pass": true },
    "historically_sequential": { "score": "3/3", "pass": true }
  },
  "failing_pillar": "epistemically_honest",
  "reason": "Confidence stated as certainty on an unverified empirical claim",
  "escalation": "Amethyst",
  "quarantine": "docs/drafts/",
  "timestamp": "YYYY-MM-DDTHH:MM:SSZ"
}
```

## Recovery Protocol

1. **Identify failing pillar** — Read `failing_pillar` and `reason` from FAIL payload
2. **Apply pillar-specific remediation:**
   - *Pillar 1 (Math):* Re-derive all quantitative claims from first principles; fix formula or remove unsupported numeric assertion
   - *Pillar 2 (Epistemic):* Add explicit confidence qualifiers; surface hidden assumptions; replace false precision with ranges
   - *Pillar 3 (Non-Violative):* Re-check against ANDROMEDA-AXIS four declarations; run OWASP Agentic Top 10 checklist; escalate to Sentinel if sovereignty boundary unclear
   - *Pillar 4 (Historical):* Trace provenance chain; identify contradicting prior artifact; either update prior artifact (with Apogee sign-off) or revise current artifact to be consistent
3. **Re-run gate** — N ≥ 3 fresh scoring runs after remediation; do not reuse prior run scores
4. **Escalation if unresolved** — After 2 full re-run cycles still failing, escalate to Njineer; artifact remains quarantined until architect decision
5. **False-positive check (P-05)** — Before logging as a BLG, confirm the failure is real vs. a scoring instrument error; log close reason explicitly in SWEEP_LOG
